# Neuroscan Frontend QnA Guide

This document details the frontend processes of the Neuroscan project from start to finish. It is tailored to help you prepare for your final year project defense, focusing on the "how" and "why" behind your technical decisions in the UI/UX layer.

---

## Part 1: Step-by-Step Breakdown of Frontend Processes

### 1. Modern UI/UX Design & Theming (CSS & Inline Styles)
*   **What was done:** Created a premium, glassmorphic design system that supports dark mode and light mode, featuring custom-built components (e.g., dropdowns, buttons, modals, floating action buttons) without relying heavily on bulky UI libraries like Bootstrap or Material-UI.
*   **How it was done:** Used pure CSS (`index.css`) for base styles, utility classes, and CSS variables for theming (`body.dark`). Component-specific layouts (like the `HospitalPortal`) utilize React inline styles with `rgba` colors and `backdrop-filter: blur()` properties to achieve the "glass" effect and smooth hover micro-animations.
*   **Why it was done:** To ensure the application feels highly responsive, modern, and lightweight. Building a custom design system gives absolute control over the look and feel, ensuring the medical tool is both aesthetically pleasing and professional.

### 2. Single Page Application (SPA) Routing & State Management (React)
*   **What was done:** Developed the interface as a true SPA, where different portals (Admin, Hospital, Patient) and screens (Landing, Pricing) are rendered dynamically without page reloads.
*   **How it was done:** Avoided external routers like `react-router-dom` in favor of a clean conditional rendering approach within `App.js`. A `currentView` state dictates which component is shown based on user authentication status (`user.type`) or URL parameters (e.g., Stripe redirects).
*   **Why it was done:** For a project of this structure, conditional state-based rendering is highly efficient, reduces bundle size, and simplifies the state flow between authentication and portal access.

### 3. Real-Time Interactions & Chat (Socket.IO-client)
*   **What was done:** Implemented real-time messaging between doctors and patients, alongside instant push notifications for when an MRI scan finishes processing.
*   **How it was done:** Used the `socket.io-client` library instantiated at the top-level of portals (like `HospitalPortal.js`). Features `useEffect` hooks to listen to events like `'new_message'` or `'notification'` and update the UI state (`notifications` array, `unreadCount`) instantly.
*   **Why it was done:** Medical settings demand immediacy. Traditional HTTP polling (requesting data every X seconds) is slow and wastes server resources. WebSockets provide a persistent, two-way connection for instant updates.

### 4. Interactive Data Visualization (Recharts)
*   **What was done:** Built analytics dashboards for admins and hospitals to track predictions, tumor distributions, and usage metrics over time.
*   **How it was done:** Integrated the `recharts` library to render responsive BarCharts, LineCharts, and AreaCharts fetching dynamic data from the backend APIs.
*   **Why it was done:** Recharts is built specifically for React, offering SVG-based, highly customizable charts that immediately respond to state changes (like switching between Daily, Weekly, or Monthly views) while perfectly adapting to light/dark themes.

### 5. Subscription Checkout Integration (Stripe Elements)
*   **What was done:** Added frontend payment flows allowing users to securely upgrade their scan limits.
*   **How it was done:** Utilized `@stripe/react-stripe-js` and `@stripe/stripe-js`. When a user opts to upgrade, the frontend requests a Checkout Session URL from the backend and performs a full-page redirect to Stripe's secure, PCI-compliant checkout page. Upon success or failure, Stripe redirects back to the SPA (`/subscription-success` or `/cancelled`).
*   **Why it was done:** To securely handle monetization without the immense legal and security burdens of processing credit card details directly on our forms.

### 6. Medical Image Validation & Pre-computation
*   **What was done:** Built an initial frontend safeguard that checks if an uploaded image is actually a valid grayscale MRI scan before sending it to the heavy PyTorch backend.
*   **How it was done:** Utilized the HTML5 `Canvas` API (`canvas.getContext("2d")`) to read image pixel data locally in the browser (checking RGB discrepancies to ensure it's grayscale, verifying contrast levels and image dimensions).
*   **Why it was done:** To save server bandwidth and expensive GPU computational cycles. If a user uploads a generic colored photo, the client catches it immediately without waking the PyTorch classification model.

## Part 2: Possible Defense Questions & Answers (Tabular Form)

| **Topic / Question** | **Expected Answer / Justification** |
| :--- | :--- |
| **Why did you use React.js instead of vanilla HTML/JS or an older framework like Angular?** | React utilizes a Virtual DOM, which makes updating the UI incredibly fast—perfect for real-time dashboards and chat systems. Its component-based architecture allowed us to build reusable elements (like CustomDropdowns or Modals) that keep the codebase modular and maintainable. |
| **Why didn't you use `react-router-dom` for navigation?** | Given the specific architecture of our role-based portals, we found that using conditional rendering (managing a `currentView` state in `App.js`) kept the auth flow tightly coupled to the UI. It prevented unauthorized access to components implicitly and reduced our dependency graph. |
| **What is Glassmorphism and how did you implement it?** | Glassmorphism is a modern UI trend mimicking frosted glass. We achieved it using CSS properties: specifically `background: rgba(255, 255, 255, 0.8)` combined with `backdrop-filter: blur(12px)`. This creates depth, making the application feel premium and modern. |
| **How does your real-time chat actually work on the frontend?** | We use `socket.io-client`. When the portal mounts, a `useEffect` hook establishes a persistent connection to the server. We attach listeners (`socket.on('new_message')`) that intercept incoming data and update the React state, triggering an instant re-render of the chat box without refreshing the page. |
| **What happens if the backend server crashes or is unreachable?** | We built a `ConnectionErrorHandler` wrapper around the App. It catches fetch failures or network timeouts and displays a friendly offline/error screen instead of breaking the entire UI or leaving the user on an infinite loading spinner. |
| **How did you handle state management? Did you use Redux?** | We opted against Redux. For this application's scale, React's built-in `useState`, `useEffect`, and passing props down the component tree was sufficient. The data flow (auth -> portal -> components) is linear enough that Redux would have added unnecessary boilerplate. |
| **How do you ensure the charts look good in both dark and light modes?** | By passing our CSS theme variables (like `textPrimary` or `bgColor` derived from our `darkMode` state) directly into Recharts' `stroke` and `fill` props, ensuring the text and grids dynamically adapt to the user's preferred theme. |
| **How does the system know when a Stripe payment is successful?** | The frontend redirects the user to Stripe. Once they pay, Stripe redirects them back to our app with a specific `session_id` in the URL. `App.js` detects this URL parameter during mapping and renders the `SubscriptionSuccess` component, while the backend verifies the payment asynchronously via Stripe Webhooks. |
